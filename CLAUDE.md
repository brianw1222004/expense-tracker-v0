@AGENTS.md

## Project

Expense-tracker demo built with Expo SDK 56 / React Native 0.85 (React 19). Plain JavaScript, no TypeScript. There is no test suite or linter; `npx expo export --platform web` is the closest thing to a build check (it catches syntax/import errors). `dist/` is the output of that export — generated, not source.

Backend is Supabase (auth + Postgres), configured via `EXPO_PUBLIC_SUPABASE_URL` / `EXPO_PUBLIC_SUPABASE_ANON_KEY` in `.env` (gitignored; template in `.env.example`, schema in `supabase/schema.sql`, one-time setup steps in `SUPABASE_SETUP.md`). **With no `.env` the app runs in local-only mode** — no sign-in screen, AsyncStorage only — so it always boots even without a backend; don't write code that assumes `supabase` is non-null (it's null when unconfigured).

## Commands

```
npm start                  # Expo dev server (LAN — phone must be on same Wi-Fi)
npx expo start --tunnel    # dev server through ngrok tunnel (works across networks)
npm run web                # run in the browser via react-native-web
npx expo export --platform web      # build check (catches syntax/import errors); regenerates dist/
node scripts/make-qr.js [exp-url]   # write a scannable QR PNG for a tunnel URL to ../expense-tracker-qr.png
```

The app is run on a phone via Expo Go (scan the QR from the terminal), or in a browser with `npm run web`. Code must work on BOTH targets — e.g. `Alert.alert` with buttons is a no-op on web (use a custom Modal or `window.confirm` fallback), and `onEndEditing` never fires on web (use `onBlur`).

## RN 0.85 gotchas (verified against the v56 docs)

- `StyleSheet.absoluteFillObject` was removed; use `StyleSheet.absoluteFill` as a style entry.
- Classic `Animated` is the supported animation API here (no reanimated). `useNativeDriver: true` only for `transform`/`opacity`; animating SVG props (`strokeDashoffset` on `Animated.createAnimatedComponent(Path)`) requires `useNativeDriver: false`. Never mix drivers on one `Animated.Value`.

## Theme, font, language — cross-cutting conventions

- **Theme is dynamic.** `src/theme.js` exports two palettes (`THEMES.cookie`, the warm-cream default, and `THEMES.midnight`, the original dark blue) plus `ThemeProvider`/`useTheme()`. Components get colors from `const { colors } = useTheme()` and build styles per theme: `const styles = useMemo(() => createStyles(colors), [colors])` with `createStyles = (colors) => StyleSheet.create({...})` at the bottom of the file. There is no static `colors` export; never import a palette directly. `spacing`/`radius`/`fonts` are still static imports. The palettes also carry `statusBarStyle`, `keyboardAppearance` (use on every TextInput), `onAccent` (text on accent-filled surfaces), semantic `success`/`warning`/`danger`, and `backdrop`.
- **Font is Caladea** (`@expo-google-fonts/caladea` + `expo-font`, loaded with `useFonts` in `App.js` before first render) — the open metric-compatible equivalent of Cambria, which is commercial and can't be bundled. Custom fonts ship one file per weight, so styles pick `fonts.regular` or `fonts.bold` and must NOT set `fontWeight` (Android mis-resolves when both are present).
- **All UI strings go through i18n** (`src/i18n.js`: en / zh-Hant / es). Components: `const t = useT()`; non-component code: `translate(language, key, vars)`. Missing keys fall back to English, then the key. Category display names are `t('cat.' + id)` — `CATEGORIES[].label` is only the English fallback. Date labels in `src/format.js` take a `language` arg and read name tables from `i18n.js` (`getDateNames`); language-dependent derived data must re-compute when `settings.language` changes (it's a dep of the `deriveViewData` memo).

## Architecture

All state lives in `App.js`; everything under `src/` is presentational or a pure helper. Navigation is hand-rolled (no react-navigation): a `tab` state (`dashboard | list | categories | account`) drives four kept-mounted tab screens (absolutely positioned, hidden via animated opacity; a Quizlet-style horizontal card-flip transition plays on each switch — `changeTab` in `App.js` drives it with `flipAnim`/`flipDirRef`/`prevTab`; dismiss the keyboard on tab change), an `addOpen` state drives the add-expense popup (`AddExpenseModal`, also kept mounted via `display: 'none'` so a half-typed expense survives dismissal), and an `overlay` state (`budget | null`) drives the budget-editor Modal sheet. When Supabase is configured and there's no session, `App.js` early-returns `AuthScreen` instead of all of this (the early return sits after every hook call — keep it that way). `ThemeProvider` + `I18nProvider` wrap both branches, fed from `settings.theme` / `settings.language`.

- `App.js` — root component. Owns `expenses` (`{id, amount, currency, note, category, createdAt}`, amount in the ENTRY currency), `settings` (`{displayCurrency, monthlyBudget, categoryBudgets, theme, language}`), `session` (Supabase auth), nav state, and `rewardNonce` (increments per added expense to trigger RewardCheck). Data flow is cache-first: on sign-in (or launch) the per-user AsyncStorage cache renders immediately, then `syncWithServer()` reconciles with Supabase in the background and on each return to foreground. A `dataUser` state records which user the in-memory data belongs to; the save-to-cache effects are gated on `dataUser === userId` so account switches can't write one user's data under another's cache key. Every mutation updates state AND enqueues a sync op (`enqueueExpenseUpsert`/`enqueueExpenseDelete`/`enqueueExpensesReplace`, `enqueueSettingsPush`) — if you add a mutation, do both or devices drift. `deriveViewData(expenses, displayCurrency, language)` computes everything the UI shows in one pass, all converted to the display currency: day-grouped sections (items gain `displayAmount`), current-month stats, and per-month aggregates (`months[].byCategory`) for the Categories tab. Category keys are normalized through `getCategory()` so stale ids group under "Other" everywhere. When the display currency changes, `updateSettings` re-denominates the stored overall budget AND every category budget via `convert()`.
- A `dayStamp` state value (today's date key, refreshed every minute and on app-resume via AppState) is a dependency of that memo so stats roll over at midnight even though `expenses` hasn't changed. Date-sensitive derived data must depend on it (`currentMonthKey` for the Categories tab is derived from it).
- **Settings sync is partial by design.** The Supabase `settings` row only has `display_currency` and `monthly_budget` columns; `theme`, `language` and `categoryBudgets` are device-local. `sync.js` never pushes the extra fields (pushing unknown columns would error and wedge the whole pending-ops queue, blocking expense sync), pulled settings are MERGED over local state (`setSettings(prev => ({...prev, ...result.settings}))` — never replaced), and `updateSettings` only enqueues a push when `displayCurrency`/`monthlyBudget` actually changed. Known limitation: a currency change made on another device doesn't re-denominate this device's local category budgets.
- `src/currency.js` — `CURRENCIES` (code/symbol/name/decimals), static `RATES_TO_USD`, and `convert(amount, from, to)`. **Conversion happens ONLY here** (single helper by design — swapping in a live-rate API is a one-function change). Amounts are stored in their entry currency forever; conversion is a display-time concern.
- `src/screens/` — one file per screen: `DashboardScreen` (summary header, budget donut chart + per-category budget bars, external spending section, empty state), `AddExpenseScreen` (the popup card rendered inside `AddExpenseModal`: amount + currency chip, category grid, note, day-stepper date with a calendar-dropdown picker; its border/background tint animates to the selected category's color; on submit it resets and the popup closes — the RewardCheck overlay is the only confirmation), `ExpenseListScreen` (filter chips + day-grouped SectionList + custom delete-confirmation Modal), `CategoriesScreen` (tab: two side-by-side donut charts comparing current month vs last month, per-category comparison cards with delta lines), `AccountScreen` (tab: account info/sign-out, live theme switcher, language switcher), `BudgetScreen` (Modal sheet: display currency, overall monthly budget, regular category budgets, external category budgets in a separate section), `AuthScreen` (email/password sign-in/sign-up, full-screen when signed out; errors render inline because `Alert` is a no-op on web).
- `src/components/` — `TabBar` (exports `TAB_BAR_HEIGHT`; four tabs + the center + button that opens the add popup; sits in-flow below the content, so screens only pad for the floating + button's overhang, not the bar height), `AddExpenseModal` (popup presenter: backdrop fade + card scale/fade, keeps children mounted while closed), `RewardCheck` (full-screen success overlay with a centered animated gradient checkmark; total on-screen time is SHOW_MS + FADE_MS — tuned to ~0.88s after two successive 30% reductions, don't lengthen without asking), `BudgetGauge` (donut chart: SVG circle with `strokeDasharray`/`strokeDashoffset` fill, zone-colored green/yellow/red; accepts an `empty` prop to render a dashed placeholder ring when no budget is set; when over budget shows a negative amount in danger color), plus `SummaryHeader` and `ExpenseRow`.
- `src/storage.js` — AsyncStorage load/save for expenses and settings, the local cache layer; both swallow errors by design (persistence is best-effort, in-memory state is the source of truth). Keys are scoped per user id; `LOCAL_USER` (`'local'`) is the unconfigured-mode sentinel and keeps the original un-suffixed keys so pre-auth installs retain their data. `loadExpenses` normalizes legacy entries (missing `currency` → USD); `loadSettings` merges `DEFAULT_SETTINGS` and guarantees a fresh `categoryBudgets` object so old caches upgrade in place.
- `src/supabase.js` — the Supabase client (null when env vars are unset) + `isSupabaseConfigured`. Auth sessions persist in AsyncStorage, NOT the expo-sqlite localStorage shim from the Expo guide — AsyncStorage also works on react-native-web. Pauses token auto-refresh while backgrounded on native.
- `src/sync.js` — offline-first sync engine. Mutations enqueue durable ops (per-user pending-ops queue in AsyncStorage, coalesced); `flush()` pushes them in order, stopping at the first failure; `syncWithServer()` = flush, then pull all rows, with `applyPendingOps()` re-applying anything still queued on top so offline edits survive a pull. Cross-device conflicts: last write to the server wins per expense row. The client never sends `user_id` — the column default (`auth.uid()`) and RLS policies (see `supabase/schema.sql`) scope every query to the signed-in user.
- `src/categories.js` — the fixed category list (id/label/emoji/color/external). Categories have an `external` flag: external categories (currently just `bills`) are tracked separately and excluded from the overall budget total on the dashboard. `REGULAR_CATEGORIES` and `EXTERNAL_CATEGORIES` are pre-filtered exports. `getCategory(id)` falls back to "Other" for unknown ids, so stored data with stale category ids still renders. Colors are mid-tone on purpose so they read on both the cream and navy themes.
- `src/format.js` — money and date formatting. `formatMoney`/`formatMoneyShort` take a currency code (symbol + decimals per currency; JPY/TWD render with 0 decimals). `dateKey()` (local-time `YYYY-MM-DD`) is the canonical day identity; `monthKey()`/`monthKeyLabel()` the month identity. Date-label functions take a `language` arg.
- `src/demoData.js` — sample expenses for the empty-state "Load demo data" button; spans three calendar months and mixes currencies so the Categories trends and conversion always have something to show.

## Screen budget

The product spec caps the app at 7 distinct views, all used: Dashboard, Add Expense (popup), Expense List, Categories (tab), Account (tab), Budget editor (sheet), Auth. Adding a view means absorbing or retiring another.
