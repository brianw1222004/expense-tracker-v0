# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Expense-tracker demo built with Expo SDK 56 / React Native 0.85 (React 19). Plain JavaScript, no TypeScript. No linter configured. `npx expo export --platform web` is the build check (catches syntax/import errors). `dist/` is the output of that export — generated, not source.

Backend is Supabase (auth + Postgres), configured via `EXPO_PUBLIC_SUPABASE_URL` / `EXPO_PUBLIC_SUPABASE_ANON_KEY` in `.env` (gitignored; template in `.env.example`, schema in `supabase/schema.sql`, one-time setup steps in `SUPABASE_SETUP.md`). **With no `.env` the app runs in local-only mode** — no sign-in screen, AsyncStorage only — so it always boots even without a backend; don't write code that assumes `supabase` is non-null (it's null when unconfigured).

## Commands

```
npm start                  # Expo dev server (LAN — phone must be on same Wi-Fi)
npx expo start --tunnel    # dev server through ngrok tunnel (works across networks)
npm run web                # run in the browser via react-native-web
npm run ios                # build & run the native iOS dev client (expo run:ios; an ios/ prebuild exists)
npm run android            # build & run the native Android dev client (expo run:android; generates android/ on first run)
npm test                   # run Jest test suite (categories, currency, format, sync, income-sync, derive, splits)
npm test -- --testPathPattern=categories   # run a single test file
npx expo export --platform web      # build check (catches syntax/import errors); regenerates dist/
node scripts/make-qr.js [exp-url]   # write a scannable QR PNG for a tunnel URL to ../expense-tracker-qr.png
```

Tests live in `src/__tests__/` with mocks in `__mocks__/` (react, react-native, async-storage, supabase-js, i18n). Jest config is in `jest.config.js`; tests run in a Node environment with babel-jest transform. When adding a new test, follow the existing pattern: use Jest's auto-injected globals (`describe`, `test`, `expect`) without importing them — require the module under test directly, and use the module-level mocks.

The app is run on a phone via Expo Go (scan the QR from the terminal), or in a browser with `npm run web`. Code must work on BOTH targets — e.g. `Alert.alert` with buttons is a no-op on web (use a custom Modal or `window.confirm` fallback), and `onEndEditing` never fires on web (use `onBlur`).

## RN 0.85 gotchas (verified against the v56 docs)

- **Read the exact versioned docs at https://docs.expo.dev/versions/v56.0.0/ before writing any code.** Expo APIs change between SDK versions; don't rely on memory or unversioned examples.

- `StyleSheet.absoluteFillObject` was removed; use `StyleSheet.absoluteFill` as a style entry.
- Classic `Animated` is the supported animation API here (no reanimated). `useNativeDriver: true` only for `transform`/`opacity`; animating SVG props (`strokeDashoffset` on `Animated.createAnimatedComponent(Path)`) requires `useNativeDriver: false`. Never mix drivers on one `Animated.Value`.

## Theme, font, language — cross-cutting conventions

- **Theme.** `src/theme.js` exports 3 palettes (`slate`, `sand`, `neutral`) plus `ThemeProvider`/`useTheme()`. `getTheme` falls back to `neutral` for unknown names. The default theme is `neutral` (grayscale). Components get colors from `const { colors } = useTheme()` and build styles per theme: `const styles = useMemo(() => createStyles(colors), [colors])` with `createStyles = (colors) => StyleSheet.create({...})` at the bottom of the file. There is no static `colors` export; never import a palette directly. `spacing`/`radius`/`fonts` are still static imports. The palette also carries `statusBarStyle`, `keyboardAppearance` (use on every TextInput), `onAccent` (text on accent-filled surfaces), semantic `success`/`warning`/`danger`, `gradientStart`/`gradientEnd` (for the Dashboard header gradient), `glowStart`/`glowEnd` (the two hues of the hero-card corner bloom, per-palette so `CardGlow` harmonises with the theme), and `backdrop`. Category emojis and colors are multi-colored (defined in `categories.js`); all other UI icons use monochrome text symbols styled with `colors.icon`.
- **Dual font system** (`@expo-google-fonts/lora` + `@expo-google-fonts/inter`, loaded with `useFonts` in `App.js` before first render). Lora (serif) is used for all text labels via `fonts.regular` (`Lora_400Regular`), `fonts.medium` (`Lora_500Medium`), `fonts.bold` (`Lora_700Bold`). Inter (a neutral grotesque sans with strong tabular figures) is used for all numeric/monetary displays via `fonts.numRegular` (`Inter_400Regular`), `fonts.numMedium` (`Inter_500Medium`) and `fonts.numBold` (`Inter_700Bold`). Styles must NOT set `fontWeight` alongside `fontFamily` (Android mis-resolves when both are present). `fonts.medium` is used selectively (dashboard header labels); most text UI uses `regular`/`bold`. All monetary amounts, stats, chart labels, budget inputs, and percentages use the `num*` variants.
- **All UI strings go through i18n** (`src/i18n.js`: language codes `en` / `zh` (Traditional Chinese, label 繁體中文) / `es` — note the key is `zh`, not `zh-Hant`). Components: `const t = useT()`; non-component code: `translate(language, key, vars)`. Missing keys fall back to English, then the key. Category display names are `t('cat.' + id)` — `CATEGORIES[].label` is only the English fallback. Date labels in `src/format.js` take a `language` arg and read name tables from `i18n.js` (`getDateNames`); language-dependent derived data must re-compute when `settings.language` changes (it's a dep of the `deriveViewData` memo).

## Architecture

All state lives in `App.js`; everything under `src/` is presentational or a pure helper. Navigation is hand-rolled (no react-navigation): a `tab` state (`dashboard | list | categories | split | balance`) drives five kept-mounted tab screens (absolutely positioned, hidden via animated opacity; a horizontal slide transition plays on each switch — `changeTab` in `App.js` drives it with `slideAnim`/`slideDirRef`/`prevTab`; the content container uses `overflow: 'hidden'` to clip off-screen screens during transitions; dismiss the keyboard on tab change). Keep `TAB_INDEX`/`TAB_NAMES` (App.js) and the `TABS` array (TabBar) in the same order. The add "+" is NOT in the tab bar — it's a floating button (`addFab` in `App.js`) pinned bottom-right; `openAdd(mode)` opens the shared add popup (`AddExpenseModal`, kept mounted via `display: 'none'` so a half-typed entry survives dismissal) — a header pill (`EntryTypeToggle`) switches an `addMode` state (`expense | income`) between the expense and income forms, both mounted inside that one popup. An `overlay` state (`budget | account | createGroup | null`) drives the budget-editor, account, and create-group Modal sheets; `activeGroupId` drives the group-detail sheet and `addSplitFor` (a group id) drives the add-bill sheet. Account is no longer a tab — it opens from a floating account button (`accountFab` in `App.js`) pinned top-left. Both FABs are global chrome gated by a single `chromeVisible` flag that hides them whenever any popup or sheet is open (`overlay`, `addOpen`, `editingExpense`, `editingIncome`, `activeGroupId`, `addSplitFor` — extend it if you add another popup/sheet). There are NO early returns inside the main component: every hook runs unconditionally, then a single `let content` is assigned by an if/else chain — `AuthScreen` when Supabase is configured and there's no session, `null` while data loads, `OnboardingScreen` until onboarding is done, else the main UI — and one `return` at the end wraps `content` in `ThemeProvider` + `I18nProvider` (fed from `settings.theme` / `settings.language`). Keep new conditional logic in the `content` assignment, and put any new top-level chrome (FABs, sheets) inside the main-UI branch so it never renders over Auth/Onboarding. (The only early `return` is the font-loading guard in the outer `App()` wrapper.)

- `App.js` — root component. Owns `expenses` (`{id, amount, currency, note, category, createdAt}`, amount in the ENTRY currency), `income` (`{id, amount, currency, source, note, createdAt}` — a parallel entity to expenses, amount in the ENTRY currency), `settings` (`{displayCurrency, monthlyBudget, categoryBudgets, customCategories, theme, language, onboardingDone}`), `session` (Supabase auth), nav state, and `rewardNonce` (increments per added expense to trigger RewardCheck). Data flow is cache-first: on sign-in (or launch) the per-user AsyncStorage cache renders immediately, then `syncWithServer()` reconciles with Supabase in the background and on each return to foreground. A `dataUser` state records which user the in-memory data belongs to; the save-to-cache effects are gated on `dataUser === userId` so account switches can't write one user's data under another's cache key. Every mutation updates state AND enqueues a sync op (`enqueueExpenseUpsert`/`enqueueExpenseDelete`/`enqueueExpensesReplace`, `enqueueIncomeUpsert`/`enqueueIncomeDelete`/`enqueueIncomeReplace`, `enqueueSettingsPush`) — if you add a mutation, do both or devices drift. Income syncs through a SEPARATE per-user queue (keyed `${userId}::income`) so a problem on one lane (e.g. the `income` table not yet created) can never wedge the other; the income pull is tolerant (errors leave income null and the cache stands). `deriveViewData(expenses, displayCurrency, language)` computes everything the UI shows in one pass, all converted to the display currency: day-grouped sections (items gain `displayAmount`), current-month stats (`monthTotal`, `lastMonthTotal`, `todayTotal`, `avgPerDay`), per-month aggregates (`months[].byCategory`) for the Categories tab, and `dailyTotals` (per-day spending for the current month, used by `SpendingChart`). Category keys are normalized through `getCategory()` so stale ids group under "Other" everywhere. When the display currency changes, `updateSettings` re-denominates the stored overall budget AND every category budget via `convert()`. Income entries (like expenses) are stored in their entry currency and converted at display time, so they need no re-denomination. When Supabase is configured and the user hasn't completed onboarding (`!settings.onboardingDone`), App.js shows `OnboardingScreen` before the main UI.
- A `dayStamp` state value (today's date key, refreshed every minute and on app-resume via AppState) is a dependency of that memo so stats roll over at midnight even though `expenses` hasn't changed. Date-sensitive derived data must depend on it (`currentMonthKey` for the Categories tab is derived from it).
- **Settings sync is partial by design.** The Supabase `settings` row only has `display_currency` and `monthly_budget` columns; `theme`, `language` and `categoryBudgets` are device-local. `sync.js` never pushes the extra fields (pushing unknown columns would error and wedge the whole pending-ops queue, blocking expense sync), pulled settings are MERGED over local state (`setSettings(prev => ({...prev, ...result.settings}))` — never replaced), and `updateSettings` only enqueues a push when `displayCurrency`/`monthlyBudget` actually changed. Known limitation: a currency change made on another device doesn't re-denominate this device's local category budgets.
- `src/currency.js` — `CURRENCIES` (code/symbol/name/decimals), static `RATES_TO_USD`, and `convert(amount, from, to)`. **Conversion happens ONLY here** (single helper by design — swapping in a live-rate API is a one-function change). Amounts are stored in their entry currency forever; conversion is a display-time concern.
- `src/screens/` — one file per screen: `DashboardScreen` (greeting row; a Monthly Spending hero card — a `CurrencyPill` (opens the shared `CurrencyPicker` page), hero total, month-over-month delta badge, month pill, embedded `SpendingChart`, and a soft iridescent upper-right corner glow via `CardGlow`, an SVG `RadialGradient` bloom (theme-aware `glowStart`→`glowEnd` hues) clipped to the card behind the content; then a collapsible budget section with `BudgetGauge` + category bars, and the empty state), `AddEntryScreen` (the SHARED add/edit popup card for BOTH expenses and income, rendered inside `AddExpenseModal`, driven by a `mode` prop — `expense | income`. The two modes are identical in layout, styling and behavior; the ONLY differences are the selector area (a paginated category selector for expense — a horizontal `ScrollView` with `pagingEnabled` showing 8 categories per page in a 4×2 grid with dot indicators — vs. a fixed source picker for income from `incomeSources.js`) and the expense-only ÷ split tool in the amount row; everything else is identical (Add Expense is the primary design and Add Income mimics its format — the save button is `accent`-colored in BOTH modes); the card's border/background tint animates to the selected category/source color in BOTH modes. When adding (not editing) the header shows a centered `EntryTypeToggle` pill — Expense | Income — that flips `addMode`; in edit mode it shows a title plus a red Delete button (both modes; expense delete wired to `deleteExpense`). Calendar date picker, numeric-only amount input, always-visible inline currency chips, note with counter; on add it resets and the popup closes — the RewardCheck overlay is the only confirmation for expenses), `ExpenseListScreen` (inline calendar for day selection, category filter chips, expense rows for the selected day, custom delete-confirmation Modal), `CategoriesScreen` (tab: a single modern wedge-segment donut of the selected month's spending by category — month-nav arrows, total + `total` caption in the center — flanked by Most-spending / Least-spending stat cards (`ExtremeStat`) that share the donut's total as denominator (the ring still carries all categories; the cards surface only the extremes); below it a draggable per-category grid shows each category's month amount and month-over-month delta. Donut geometry lives in `buildArcs` (each segment an annular sector with straight radial butt-capped edges, arc length proportional to its share of the circumference left after a fixed gap is reserved per segment). `AddCategoryModal` inside this file provides paginated icon selection — 14 icons per page in a 7×2 swipeable grid — plus 18 preset colors and a custom hue-slider color picker using SVG gradients and PanResponder), `IncomeBalanceScreen` (tab, 4th slot: an all-time Balance hero = Σincome − Σexpenses with a month-over-month delta; a hand-rolled tappable grouped income/expense bar chart (Views, classic `Animated`, no chart lib) with a 3M/6M toggle — tapping a month refocuses the hero and "Back to current" resets it; income entries grouped by month with per-month totals; long-press a row to delete, tap to edit (income is added via the shared add popup — the tab-bar "+" toggled to "Income" — not a dedicated FAB). Per-month/all-time expense totals come from App.js (`expenseByMonth`/`totalExpenses`, derived from `months`); income aggregates are computed in-screen from the raw `income` prop), `AccountScreen` (near-full ~92% bottom Modal sheet opened by the floating account button, not a tab: account info/sign-out, language + theme switcher; mirrors `BudgetScreen`'s present/dismiss mechanism with an added grab handle), `BudgetScreen` (Modal sheet: display currency, overall monthly budget, regular category budgets, external category budgets in a separate section), `AuthScreen` (email/password sign-in/sign-up, full-screen when signed out; errors render inline because `Alert` is a no-op on web), `OnboardingScreen` (first-run setup: monthly budget + language; shown only when Supabase is configured and `onboardingDone` is false).
- `src/components/` — `TabBar` (exports `TAB_BAR_HEIGHT`; position absolute at the bottom with frosted glass blur via `expo-blur`; four tab icons + center "+" button; screens must add `paddingBottom: TAB_BAR_HEIGHT` to their ScrollView contentContainerStyle so content isn't hidden behind it), `AddExpenseModal` (popup presenter: backdrop fade + card scale/fade, keeps children mounted while closed), `EntryTypeToggle` (segmented Expense/Income pill shown in the add popup header when adding; the active segment is tinted with the `accent` color in both modes; both add screens render it and call back to `setAddMode`), `RewardCheck` (full-screen success overlay with a centered animated gradient checkmark; total on-screen time is SHOW_MS + FADE_MS — tuned to ~0.88s after two successive 30% reductions, don't lengthen without asking), `SpendingChart` (SVG line chart: smoothed cubic Bezier path of daily totals with gradient fill, peak marker, dashed horizontal grid lines with y-axis labels, x-axis day labels; accepts `quickStats` prop for inline stats header), `BudgetGauge` (donut chart: two butt-capped (straight radial-edged) `strokeLinecap` segments separated by a gap — spent (zone-colored green/yellow/red) and remaining (a neutral light-gray track, `border`) — built by `buildGaugeArcs`, mirroring the CategoriesScreen donut; a fully-spent/over ring collapses to a single segment; accepts an `empty` prop to render a dashed placeholder ring when no budget is set; when over budget shows a negative amount in danger color), plus `ExpenseRow`.
- `src/storage.js` — AsyncStorage load/save for expenses, income and settings, the local cache layer; all swallow errors by design (persistence is best-effort, in-memory state is the source of truth). Keys are scoped per user id; `LOCAL_USER` (`'local'`) is the unconfigured-mode sentinel and keeps the original un-suffixed keys so pre-auth installs retain their data. `loadExpenses`/`loadIncome` normalize legacy entries (missing `currency` → USD; income also defaults `source`/`note`); `loadSettings` merges `DEFAULT_SETTINGS` and guarantees a fresh `categoryBudgets` object so old caches upgrade in place.
- `src/supabase.js` — the Supabase client (null when env vars are unset) + `isSupabaseConfigured`. Auth sessions persist in AsyncStorage, NOT the expo-sqlite localStorage shim from the Expo guide — AsyncStorage also works on react-native-web. Pauses token auto-refresh while backgrounded on native.
- `src/sync.js` — offline-first sync engine. Mutations enqueue durable ops (per-lane pending-ops queue in AsyncStorage, coalesced); `flush()`/`flushIncome()` push them in order, stopping at the first failure; `syncWithServer()` = flush both lanes, then pull all rows, with `applyPendingOps()`/`applyPendingIncomeOps()` re-applying anything still queued on top so offline edits survive a pull. **Expenses and income use SEPARATE queues** (keys `userId` and `${userId}::income`) sharing the same generic load/persist/flush helpers (`flushQueue`), so one lane can't wedge the other; the income pull is tolerant (a missing `income` table or any error returns `income: null` and the cache stands). Cross-device conflicts: last write to the server wins per row. The client never sends `user_id` — the column default (`auth.uid()`) and RLS policies (see `supabase/schema.sql`) scope every query to the signed-in user.
- `src/categories.js` — 8 built-in categories (`CATEGORIES`) plus user-created custom categories stored in `settings.customCategories` (device-local). `external` flag separates bills-like categories from budget tracking. Every function takes `customCategories` as a parameter (not module-level state): `getAllCategories(customCategories)` / `getRegularAll(customCategories)` / `getExternalAll(customCategories)` return merged lists; `getCategory(id, customCategories)` searches both and falls back to "Other". `getCategoryLabel(category, t)` resolves display names (i18n for built-in, raw label for custom). `generateCategoryId()` creates a unique id for new custom categories. `EMOJI_OPTIONS` (24 icons, displayed in paginated pages of 14) and `COLOR_OPTIONS` (18 preset colors) are curated palettes for the add-category picker. Screens receive merged category lists as props from App.js; built-in categories cannot be deleted.
- `src/incomeSources.js` — 6 fixed income sources (`INCOME_SOURCES`: salary/freelance/side_income/reimbursement/gift/other), each with a muted color dot. No user-created sources. `getIncomeSource(id)` falls back to "Other"; `getIncomeSourceLabel(source, t)` resolves the i18n name (`t('source.' + id)`). Mirrors `categories.js`'s shape.
- `src/format.js` — money and date formatting. `formatMoney`/`formatMoneyShort` take a currency code (symbol + decimals per currency; JPY/TWD render with 0 decimals). `dateKey()` (local-time `YYYY-MM-DD`) is the canonical day identity; `monthKeyLabel()` formats a `YYYY-MM` key as a human-readable month/year string. `buildCalendarWeeks(year, month)` returns a grid of 7-cell rows for calendar UIs (shared by AddEntryScreen and ExpenseListScreen). Date-label functions take a `language` arg.
- `src/icons.js` — registry wrapper around `@hugeicons/react-native`; maps string keys (e.g. `'hamburger-01'`) to imported icon components. `HIcon` is the single render entry point. Category items and all UI chrome use this; never import icon components directly. `HIcon` forwards the `style` prop directly to `HugeiconsIcon` (both the matched and fallback branches pass it through).
- `src/derive.js` — `deriveViewData(expenses, displayCurrency, language, customCategories, now, extraSpending)` — single-pass derivation of all display data (day sections, month stats, category totals, daily chart). Called as a memo in App.js. `extraSpending` (from `yourShareAsExpenses`) folds split-bill shares into aggregates without adding them to the Expenses list sections. `now` is injectable for deterministic tests.
- `src/demoData.js` — sample expenses (`buildDemoExpenses`) and income (`buildDemoIncome`) for the empty-state "Load demo data" button; both span three calendar months and mix currencies so the Categories trends, the Income/Balance chart, and conversion always have something to show.

The original product spec capped the app at 10 distinct views; the Split Bills feature deliberately expanded that. Current views: Dashboard, Add Expense (popup), Expense List, Categories (tab), Split Bills (tab), Income/Balance (tab), Add Income (popup), Group detail (sheet), Create group (sheet), Add bill (sheet), Account (sheet, opened by the floating account button), Budget editor (sheet), Auth, Onboarding (first-run only). Add Expense and Add Income are the SAME component (`AddEntryScreen`) rendered in two modes — they count as two views but share one file. The tab bar now holds five tabs (Dashboard, Expenses, Categories, Split, Balance) and the add "+" moved out of the tab bar to a bottom-right floating button; the Add Income popup reuses the `AddExpenseModal` presenter and is reached by toggling the shared add popup's header pill to "Income" (there is no separate income FAB).

## Split Bills

A Splitwise-style ledger synced to Supabase via two separate lanes (`groups` and `split_expenses` tables, queue keys `${userId}::groups` and `${userId}::splits`). Pulls are tolerant like income — if the tables don't exist yet, the local cache stands. `src/splits.js` is the pure domain layer: the `YOU` owner sentinel, `PAYMENT_METHODS`/`getPaymentMethod`/`getPaymentMethodLabel`, `computeShares` (equal/custom, currency-decimal aware), `customSharesValid`, `groupBalances`/`groupNet`/`overallBalance` (a personal ledger — only debts involving YOU; positive = a member owes you), and `yourShareAsExpenses`. Groups hold members (typed names, no accounts), a `currency`, and a `paymentMethod` (cash/card/bank/mobile/other) — there is NO group "type/category". `splitExpenses` hold shared bills (`{id, groupId, description, amount, currency, category, paidBy, mode, shares, createdAt}`) and settlement records (`{settlement: true, from, to, ...}`). Local cache: `loadGroups`/`saveGroups`/`loadSplitExpenses`/`saveSplitExpenses` in `storage.js`; sync: `enqueueGroupUpsert`/`enqueueGroupDelete`/`enqueueSplitUpsert`/`enqueueSplitDelete` in `sync.js`. App.js owns `createGroup`/`updateGroup` (edit currency/payment method in place)/`deleteGroup`/`addSplitExpense`/`deleteSplitExpense`/`settleUp`. Screens: `SplitBillsScreen` (tab, centered title), `GroupDetailScreen` (a Group-settings card with an editable currency pill + payment-method chips), `CreateGroupScreen`, `AddSplitScreen` (sheets). **"Your share counts as spending":** `yourShareAsExpenses(splitExpenses)` synthesizes expense-like items for your share of each bill, passed to `deriveViewData` as its 6th `extraSpending` arg — these fold into every spending aggregate (month total, category totals, daily chart, per-month breakdown, budgets) but are deliberately kept OUT of the Expenses list `sections` (so the list stays direct-expenses-only). The Dashboard shows a split-balances widget; balances display in each group's own currency, the overall summary converts to the display currency.
