# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Expense-tracker demo built with Expo SDK 56 / React Native 0.85 (React 19). Plain JavaScript, no TypeScript. No linter configured. `npx expo export --platform web` is the build check (catches syntax/import errors). `dist/` is the output of that export — generated, not source.

Backend is Supabase (auth + Postgres), configured via `EXPO_PUBLIC_SUPABASE_URL` / `EXPO_PUBLIC_SUPABASE_ANON_KEY` in `.env` (gitignored; template in `.env.example`, schema in `supabase/schema.sql`, one-time setup steps in `SUPABASE_SETUP.md`). **With no `.env` the app runs in local-only mode** — no sign-in screen, AsyncStorage only — so it always boots even without a backend; don't write code that assumes `supabase` is non-null (it's null when unconfigured).

## Commands

```
npm start                  # Expo dev server (LAN — phone must be on same Wi-Fi)
npx expo start --tunnel    # dev server through ngrok tunnel (works across networks)
npm run web                # run in the browser via react-native-web
npm run ios                # build & run the native iOS dev client (expo run:ios; an ios/ prebuild exists)
npm run android            # build & run the native Android dev client (expo run:android; generates android/ on first run)
npm test                   # run Jest test suite (10 suites in src/__tests__/, incl. splits-regression, sync-groups-splits, storage, budget)
npm test -- --testPathPatterns=categories  # run a single test file (note: plural --testPathPatterns)
npx expo export --platform web      # build check (catches syntax/import errors); regenerates dist/
node scripts/make-qr.js [exp-url]   # write a scannable QR PNG for a tunnel URL to ../expense-tracker-qr.png
```

Tests live in `src/__tests__/` with mocks in `__mocks__/` (react, react-native, async-storage, supabase-js, i18n). Jest config is in `jest.config.js`; tests run in a Node environment with babel-jest transform. When adding a new test, follow the existing pattern: use Jest's auto-injected globals (`describe`, `test`, `expect`) without importing them — require the module under test directly, and use the module-level mocks.

The app is run on a phone via Expo Go (scan the QR from the terminal), or in a browser with `npm run web`. Code must work on BOTH targets — e.g. `Alert.alert` with buttons is a no-op on web (use a custom Modal or `window.confirm` fallback), and `onEndEditing` never fires on web (use `onBlur`).

## RN 0.85 gotchas (verified against the v56 docs)

- **Read the exact versioned docs at https://docs.expo.dev/versions/v56.0.0/ before writing any code.** Expo APIs change between SDK versions; don't rely on memory or unversioned examples.

- `StyleSheet.absoluteFillObject` was removed; use `StyleSheet.absoluteFill` as a style entry.
- Classic `Animated` is the supported animation API here (no reanimated). `useNativeDriver: true` only for `transform`/`opacity`; animating SVG props (`strokeDashoffset` on `Animated.createAnimatedComponent(Path)`) requires `useNativeDriver: false`. Never mix drivers on one `Animated.Value`.

## Theme, font, language — cross-cutting conventions

- **Theme.** `src/theme.js` exports 3 palettes (`slate`, `sand`, `neutral`) plus `ThemeProvider`/`useTheme()`. `getTheme` falls back to `neutral` for unknown names. The default theme is `neutral` (grayscale). Components get colors from `const { colors } = useTheme()` and build styles per theme: `const styles = useMemo(() => createStyles(colors), [colors])` with `createStyles = (colors) => StyleSheet.create({...})` at the bottom of the file. There is no static `colors` export; never import a palette directly. `spacing`/`radius`/`fonts` are still static imports, as are the `cardShadow`/`panelShadow` elevation tokens (spread into a style object instead of re-declaring per-file shadow props). The palette also carries `statusBarStyle`, `keyboardAppearance` (use on every TextInput), `onAccent` (text on accent-filled surfaces), semantic `success`/`warning`/`danger`, `glowStart`/`glowEnd`/`glowWashTop` (the two hues of the top-of-page wash every tab screen renders — the shared `HeaderGlow` component — plus the pre-blended solid App.js paints on the status-bar strip while the tabbed UI is visible; `glowWashTop` must stay in sync with the wash's 0.34 top stop opacity), and `backdrop`. Category emojis and colors are multi-colored (defined in `categories.js`); all other UI icons use monochrome text symbols styled with `colors.icon`.
- **Unified font system** (Liberation Sans family, bundled as TTFs in `assets/fonts/` with their OFL licenses, loaded with `useFonts` from `expo-font` in `App.js` before first render). One design drives BOTH text and numbers, so word labels align with the money/stat figures beside them. All non-bold text is slightly bolded by design: `fonts.regular`/`numRegular` resolve to `Arimo-Medium` (Arimo is the Google-Fonts continuation of Liberation Sans — metric-identical — supplying the 500 weight the Liberation family lacks), while `fonts.medium`/`bold`/`numMedium`/`numBold` all resolve to `LiberationSans-Bold` (num* styles additionally set `fontVariant: ['tabular-nums']`). Styles must NOT set `fontWeight` alongside `fontFamily` (Android mis-resolves when both are present). All monetary amounts, stats, chart labels, budget inputs, and percentages use the `num*` variants. (Inter and Lora were retired; nothing imports `@expo-google-fonts/*` any more.)
- **All UI strings go through i18n** (`src/i18n.js`: language codes `en` / `zh` (Traditional Chinese, label 繁體中文) / `es` — note the key is `zh`, not `zh-Hant`). Components: `const t = useT()`; non-component code: `translate(language, key, vars)`. Missing keys fall back to English, then the key. Category display names are `t('cat.' + id)` — `CATEGORIES[].label` is only the English fallback. Date labels in `src/format.js` take a `language` arg and read name tables from `i18n.js` (`getDateNames`); language-dependent derived data must re-compute when `settings.language` changes (it's a dep of the `deriveViewData` memo).

## Architecture

All state lives in `App.js`; everything under `src/` is presentational or a pure helper. Navigation is hand-rolled (no react-navigation): a `tab` state (`dashboard | list | split | insight`) drives four kept-mounted tab screens (absolutely positioned, hidden via animated opacity; a Copilot-style widgets-swap transition plays on each switch: a fixed backdrop layer in App.js — the same background + `HeaderGlow` wash every tab paints — stays put while the screens crossfade over it with a small directional glide (`COPILOT_GLIDE`), so only the widgets appear to change; the `src/useTabSlide.js` hook owns the whole transition (`tab`/`prevTab`/`slideAnim`/`slideDirRef` + the swipe `PanResponder`) and returns `{ tab, changeTab, screenStyle, swipeHandlers }` to App.js — `changeTab` drives a tap switch, `swipeHandlers` (spread on the content View) drives swipe-to-switch (the old full-page horizontal-slide transition was removed); the content container uses `overflow: 'hidden'` to clip off-screen screens during transitions; dismiss the keyboard on tab change). Keep `TAB_INDEX`/`TAB_NAMES` (`src/useTabSlide.js`) and the `TABS` array (TabBar) in the same order. The add "+" is the center button of the tab bar (TabBar takes an `onAdd` prop); `openAdd()` opens the add popup (`AddExpenseModal`, kept mounted via `display: 'none'` so a half-typed entry survives dismissal; because it's a plain View — not an RN Modal — App.js installs an Android `BackHandler` that closes the add/edit popups on hardware back, while the Sheet-based overlays are real Modals and handle back themselves). The add popup has a **Personal/Shared** header toggle (`EntryModeToggle`, driven by `addEntryMode` in App.js): Personal renders `AddEntryScreen` (add an expense), Shared renders `SharedSplitForm` (add a split bill). An `addNonce` state keys the Shared form so it remounts fresh on each open. An `overlay` state (`budget | account | createGroup | null`) drives the budget-editor, account, and create-group Modal sheets; `activeGroupId` drives the group-detail sheet. A group's "Add a bill" reuses the Shared add popup locked to that group (`sharedLockedGroupId`, set by `openSharedAddForGroup`; `closeAdd` reopens the group sheet afterward). Every tab shows a centered ‹ month › selector (the shared `MonthSelector` component) under its page title, and each tab's month is INDEPENDENT — changing the month on one page never affects another. Dashboard's lives in App.js (`dashMonthKey`/`shiftDashMonth`, because `heroView` derives there) and scopes the hero card + category summary card; Expenses, Insight and Split own theirs as local screen state (kept-mounted tabs preserve it). Split's is deliberately display-only — balances are outstanding debts and stay all-time. The category card's "more detail" pill navigates to the Insight tab (`changeTab('insight')`), which hosts the full per-category row list on its Categories card (there is no separate breakdown page any more). Account is no longer a tab — it opens from a floating account button (`accountFab` in `App.js`) pinned top-left, the only floating FAB (the add "+" lives in the tab bar). It's gated by a single `chromeVisible` flag that hides it whenever any popup or sheet is open (`overlay`, `addOpen`, `editingExpense`, `activeGroupId` — extend it if you add another popup/sheet). There are NO early returns inside the main component: every hook runs unconditionally, then a single `let content` is assigned by an if/else chain — `AuthScreen` when Supabase is configured and there's no session, `null` while data loads, `OnboardingScreen` until onboarding is done, else the main UI — and one `return` at the end wraps `content` in `ThemeProvider` + `I18nProvider` (fed from `settings.theme` / `settings.language`). Keep new conditional logic in the `content` assignment, and put any new top-level chrome (FABs, sheets) inside the main-UI branch so it never renders over Auth/Onboarding. (The only early `return` is the font-loading guard in the outer `App()` wrapper.)

- `App.js` — root component. Owns `expenses` (`{id, amount, currency, note, category, createdAt}`, amount in the ENTRY currency), `settings` (`{displayCurrency, monthlyBudget, categoryBudgets, customCategories, theme, language, onboardingDone}`), `session` (Supabase auth), nav state, and `rewardNonce` (increments per added expense to trigger RewardCheck). Data flow is cache-first: on sign-in (or launch) the per-user AsyncStorage cache renders immediately, then `syncWithServer()` reconciles with Supabase in the background and on each return to foreground. A `dataUser` state records which user the in-memory data belongs to; the save-to-cache effects are gated on `dataUser === userId` so account switches can't write one user's data under another's cache key. Every mutation updates state AND enqueues a sync op (`enqueueExpenseUpsert`/`enqueueExpenseDelete`/`enqueueExpensesReplace`, `enqueueSettingsPush`) — if you add a mutation, do both or devices drift. `deriveViewData(expenses, displayCurrency, language)` computes everything the UI shows in one pass, all converted to the display currency: day-grouped sections (items gain `displayAmount`), current-month stats (`monthTotal`, `lastMonthTotal`), per-month aggregates (`months[].byCategory`, plus `months[].dailyTotals` — each month's own per-day series, feeding the hero chart for whichever month is selected) for the Categories tab, and `dailyTotals` (per-day spending for the current month, used by `SpendingChart`). Category keys are normalized through `getCategory()` so stale ids group under "Other" everywhere. When the display currency changes, `updateSettings` re-denominates the stored overall budget AND every category budget via `convert()`. When Supabase is configured and the user hasn't completed onboarding (`!settings.onboardingDone`), App.js shows `OnboardingScreen` before the main UI.
- A `dayStamp` state value (today's date key, refreshed every minute and on app-resume via AppState) is a dependency of that memo so stats roll over at midnight even though `expenses` hasn't changed. Date-sensitive derived data must depend on it (`currentMonthKey` for the Categories tab is derived from it).
- **Settings sync.** The Supabase `settings` row carries `displayCurrency`, `monthlyBudget`, `categoryBudgets`, `categoryOrder`, `customCategories`, `customPaymentMethods` (see `pickSyncedSettings`/`toSettingsRow`/`fromSettingsRow` in `sync.js`); only per-device preferences (`theme`, `language`, `onboardingDone`) stay device-local. The extra columns are NULLABLE — NULL means "no device pushed this yet" and `fromSettingsRow` omits it, so the merge over local (`setSettings(prev => ({...prev, ...result.settings}))` — never replaced) can't clobber a local value with an empty default. `updateSettings` only enqueues a push (of the whole synced subset) when a synced field actually changed; device-local-only patches must not bump `settingsVersionRef`. The settings push falls back to the legacy two-column payload on an unknown-column error (database not yet migrated with `supabase/migrate-settings-sync.sql`), so it can never wedge the expense lane; App.js seeds the new columns from local values on the first pull that finds them unset. Because `categoryBudgets` sync with the currency, a currency change on another device now arrives already re-denominated.

- `src/storage.js` — AsyncStorage load/save for expenses and settings, the local cache layer; all swallow errors by design (persistence is best-effort, in-memory state is the source of truth). Keys are scoped per user id; `LOCAL_USER` (`'local'`) is the unconfigured-mode sentinel and keeps the original un-suffixed keys so pre-auth installs retain their data. `loadExpenses` normalizes legacy entries (missing `currency` → USD); `loadSettings` merges `DEFAULT_SETTINGS` and guarantees a fresh `categoryBudgets` object so old caches upgrade in place. `clearUserStorage` also wipes legacy keys from retired features (category order, income).
- `src/supabase.js` — the Supabase client (null when env vars are unset) + `isSupabaseConfigured`. Auth sessions persist in AsyncStorage, NOT the expo-sqlite localStorage shim from the Expo guide — AsyncStorage also works on react-native-web. Pauses token auto-refresh while backgrounded on native.
- `src/sync.js` — offline-first sync engine. Mutations enqueue durable ops (per-lane pending-ops queue in AsyncStorage, coalesced); `flush()`/`flushGroups()`/`flushSplits()` push them in order, stopping at the first failure; `syncWithServer()` = flush all lanes, then pull all rows, with `applyPendingOps()`/`applyPendingGroupOps()`/`applyPendingSplitOps()` re-applying anything still queued on top so offline edits survive a pull. **Expenses/settings, groups, and splits use SEPARATE queues** (keys `userId`, `${userId}::groups`, `${userId}::splits`) sharing the same generic load/persist/flush helpers (`flushQueue`), so one lane can't wedge the others; the groups/splits pulls are tolerant (a missing table or any error returns null for that field and the cache stands). Cross-device conflicts: last write to the server wins per row. The client never sends `user_id` — the column default (`auth.uid()`) and RLS policies (see `supabase/schema.sql`) scope every query to the signed-in user.
- `src/categories.js` — 8 built-in categories (`CATEGORIES`) plus user-created custom categories stored in `settings.customCategories` (synced via the settings row). `external` flag separates bills-like categories from budget tracking. Every function takes `customCategories` as a parameter (not module-level state): `getAllCategories(customCategories)` / `getRegularAll(customCategories)` / `getExternalAll(customCategories)` return merged lists; `getCategory(id, customCategories)` searches both and falls back to "Other". `getCategoryLabel(category, t)` resolves display names (i18n for built-in, raw label for custom). `generateCategoryId()` creates a unique id for new custom categories. `EMOJI_OPTIONS` (24 icons, displayed in paginated pages of 14) and `COLOR_OPTIONS` (18 preset colors) are curated palettes for the add-category picker. Screens receive merged category lists as props from App.js; built-in categories cannot be deleted.
- `src/format.js` — money and date formatting. `formatMoney`/`formatMoneyShort` take a currency code (symbol + decimals per currency; JPY/TWD render with 0 decimals). `dateKey()` (local-time `YYYY-MM-DD`) is the canonical day identity; `monthKeyLabel()` formats a `YYYY-MM` key as a human-readable month/year string. `buildCalendarWeeks(year, month)` returns a grid of 7-cell rows for calendar UIs (shared by `CalendarField` and `ExpenseListScreen`). `cleanAmountInput(text)` strips everything but digits and decimal separators for the add forms' amount/share inputs. Date-label functions take a `language` arg.
- `src/budget.js` — pure budget-allocation math for the budget editor: ratio↔amount conversion against the overall budget, 5% slider snapping (`snapRatioToStep`), and clamping so the sum of category budgets never exceeds the overall (`clampCategoryBudgetAmount`/`maxBudgetForCategory`; `fitAllocatedBudgetsToOverall` proportionally shrinks existing allocations when the overall budget drops).
- `src/confirm.js` — `confirmDestructive({ title, body, confirmLabel, cancelLabel })`: the one web/native confirm helper (native `Alert.alert`, `window.confirm` on web), returning a `Promise<boolean>`. Used by every destructive action (delete expense/bill/group, settle, sign out, delete account, load demo) instead of re-implementing the Alert/web fallback inline.
- `src/icons.js` — registry wrapper around `@hugeicons/react-native`; maps string keys (e.g. `'hamburger-01'`) to imported icon components. `HIcon` is the single render entry point. Category items and all UI chrome use this; never import icon components directly. `HIcon` forwards the `style` prop directly to `HugeiconsIcon` (both the matched and fallback branches pass it through).
- `src/derive.js` — `deriveViewData(expenses, displayCurrency, language, customCategories, now, extraSpending)` — single-pass derivation of all display data (day sections, month stats, category totals, daily chart). Called as a memo in App.js. `extraSpending` (from `yourShareAsExpenses`) folds split-bill shares into aggregates without adding them to the Expenses list sections. `now` is injectable for deterministic tests.
- `src/demoData.js` — sample expenses (`buildDemoExpenses`) for the empty-state "Load demo data" button; they span three calendar months and mix currencies so the category trends and conversion always have something to show.

## Split Bills

A Splitwise-style ledger synced to Supabase via two separate lanes (`groups` and `split_expenses` tables, queue keys `${userId}::groups` and `${userId}::splits`). Pulls are tolerant — if the tables don't exist yet, the local cache stands. `src/splits.js` is the pure domain layer: the `YOU` owner sentinel, `PAYMENT_METHODS`/`getPaymentMethod`/`getPaymentMethodLabel`/`getPaymentMethodColor`/`getAllPaymentMethods`, `computeShares` (equal/custom/percentage, currency-decimal aware, shares always reconciled to sum to the bill), `customSharesValid`/`percentageSharesValid`, `computeTaxShares` + `taxInputValid` (the itemized tax split — per-person subtotals plus a tax%/tip% distributed proportionally), `groupBalances`/`groupNet`/`overallBalance` (a personal ledger — only debts involving YOU; positive = a member owes you), `yourShareAsExpenses`, `removeMemberFromBill` + `billUndistributed` (member-removal share handling — see below), plus `GROUP_ICONS`/`DEFAULT_GROUP_ICON` (emoji avatars) and `PAYMENT_ICON_OPTIONS`. **Payment methods are category-like**: each built-in AND custom method carries a `color` + a hugeicon `icon` (rendered as an icon+color chip and used to theme a group's surface). Built-ins are fixed (cash/card/bank/mobile/other); users add custom methods via `PaymentMethodModal` (name + icon + color/hue picker, mirroring the custom-category editor) — stored in `settings.customPaymentMethods` (synced via the settings row). `getPaymentMethod`/`getPaymentMethodColor` fall back to "cash" for unknown ids, and `getPaymentMethodLabel` likewise falls back to the cash label (never echoes a raw id). Groups hold members (typed names, no accounts), a `currency`, a `paymentMethod`, and an `icon` (emoji avatar, device-local, picked via the `IconPickerSheet` page) — there is NO group "type/category". `splitExpenses` hold shared bills (`{id, groupId, description, amount, currency, category, paidBy, mode, shares, createdAt}`, plus an optional device-local `meta` holding the raw percentage/tax inputs so an edit can reconstruct them losslessly) and settlement records (`{settlement: true, from, to, ...}`). **`group.icon` and bill `meta` are device-local** (not in the Supabase `groups`/`split_expenses` column mappings); App.js's sync pull-merge preserves them by id across reconciles, same posture as theme/language. Local cache: `loadGroups`/`saveGroups`/`loadSplitExpenses`/`saveSplitExpenses` in `storage.js`; sync: `enqueueGroupUpsert`/`enqueueGroupDelete`/`enqueueSplitUpsert`/`enqueueSplitDelete` in `sync.js`. App.js owns `createGroup`/`updateGroup` (edit currency/payment method/icon/members in place)/`removeGroupMember`/`deleteGroup`/`addSplitExpense`/`updateSplitExpense`/`deleteSplitExpense`/`settleUp`/`addCustomPaymentMethod`/`removeCustomPaymentMethod` (reassigns affected groups to cash). **Member removal is bill-aware** (the × in `GroupDetailScreen`): unreferenced members are removed outright; a member who PAID a bill can't be removed (edit/delete those bills first — `alertInfo` in `confirm.js` explains); a share-holder gets a choice popup (`OptionPicker` with a `subtitle`) — **redistribute** (their share is re-split among the remaining participants via `removeMemberFromBill`: equal bills stay equal, weighted bills scale proportionally) or **reassign manually** ('unassign': the share key is dropped, the bill becomes a 'custom' split with an undistributed residual — shown on the bill row via `billUndistributed`, and with exactly one affected bill the editor opens on it immediately). Their settlements are deleted with them. Committing a member-name blur never removes anyone (an emptied name reverts). Screens: `SplitBillsScreen` (tab, centered title with a `MonthSelector` under it that is DELIBERATELY display-only — balances are outstanding debts, so the summary and group cards stay all-time regardless of the selected month (product decision; don't wire it into balance math without asking); groups render as a stack of full-width widget cards mirroring the group sheet's hero — a header row (method-tinted icon, name, members · method, overlapping avatar stack), the group's all-time total spent + toned net line (both in the display currency), then icon-badged recent-bill preview rows with toned you-lent/you-borrowed captions (`yourBillPosition` in `splits.js`, shared with the sheet) on a nested solid surface — each card washed in its payment-method color with a deeper colored left edge, the expense-row treatment), `GroupDetailScreen` (a method-tinted HERO card leads the sheet — all-time total spent in the group currency, a toned net-position line, and an overlapping avatar stack (you in the theme accent, members in their stable `memberColor` palette colors — `MEMBER_COLORS`/`memberColor(id)` in `splits.js`, also used for the member rows' avatars) — then the combined members+balances card, then the bill list (each row leads with a category-tinted icon badge and carries a toned you-lent/you-borrowed caption under the amount; needs `customCategories` to resolve custom-category icons); the payment method is a compact widget pill in the title row beside the currency pill — method label + flip chevron on the animated method-color tint — that expands into the payment-method-THEMED chip panel under the header (animated border/wash tint to the method's color, mirroring the add-expense category tint; height/opacity-animated accordion) and collapses back after a pick; the avatar is a tappable emoji; cards use a `cardShadowWrap` so `overflow:'hidden'` doesn't suppress the iOS shadow), `CreateGroupScreen` (sheets); bills are added through the Shared add popup (`SharedSplitForm`) and EDITED by tapping a bill row (`SharedSplitForm` in `editBill` mode — re-decide paid-by / split method / shares; reuses the group-sheet round-trip via `editingSplit` in App.js). **"Your share counts as spending":** `yourShareAsExpenses(splitExpenses)` synthesizes expense-like items for your share of each bill, passed to `deriveViewData` as its 6th `extraSpending` arg — these fold into every spending aggregate (month total, category totals, daily chart, per-month breakdown, budgets) but are deliberately kept OUT of the Expenses list `sections` (so the list stays direct-expenses-only). The Dashboard shows a split-balances widget; balances display in each group's own currency, the overall summary converts to the display currency.
